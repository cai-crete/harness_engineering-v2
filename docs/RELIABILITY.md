# RELIABILITY.md — 안정성 기준 문서

**관장하는 영역:** 시스템이 예상치 못한 상황에서 어떻게 동작해야 하는가

에이전트가 새 기능을 구현하거나 기존 코드를 수정할 때, 에러 처리, 폴백, 재시도, 데이터 정합성 등의 안정성 관련 코드를 어떻게 작성해야 하는지의 기준을 제공합니다.

**다루는 내용:**

- 어떤 종류의 에러를 어떻게 처리하는지 (복구 가능 vs 복구 불가능)
- 외부 서비스가 실패했을 때 어떤 폴백 전략을 사용하는지
- 재시도 정책 — 언제 재시도하고 언제 실패를 전파하는지
- 로깅 기준 — 무엇을, 어떤 레벨로 기록하는지
- 데이터 정합성 보장 방식
- 시스템 경계에서의 유효성 검증 기준

## 알려진 공통 실패 모드

| 실패 모드 | 증상 | 처방 |
|-----------|------|------|
| **Protocol 무시** | 출력이 Action Protocol 구조와 무관 | `system` 파라미터 실제 삽입 여부 코드 레벨 확인 |
| **Knowledge Doc 충돌** | Protocol과 Knowledge Doc 지시가 상충 | Protocol이 우선함을 system prompt 상단에 명시 |
| **반복 루프** | 동일 출력이 계속 반복 | Pre-Step에 "이전 출력과 다른 관점 강제" 조건 추가 |

> Protocol 검증 파이프라인 및 오염 감지 대응 절차: `docs/design-docs/protocol-design-guide.md §7, §8`

## 시스템 레벨 신뢰성

### API 안정성

| 항목 | 기준 |
|------|------|
| API 응답 타임아웃 | 30초 이내 |
| 재시도 정책 | 최대 2회, 지수 백오프 |
| 에러 응답 처리 | 사용자에게 명확한 오류 메시지 표시 |
| Protocol 주입 실패 | `system` 파라미터 null 방어 코드 필수 |

### Protocol 파일 무결성

| 항목 | 기준 |
|------|------|
| 인코딩 | UTF-8 |
| 최대 토큰 | 시스템 프롬프트 + 지식문서 합계 모델 컨텍스트 25% 이내 |
| 버전 파일 보존 | 전 버전 삭제 금지 |
