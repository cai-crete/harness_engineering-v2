# PRODUCT_SENSE.md — 제품 판단 기준 문서

**관장하는 영역:** 모호한 상황에서 어떤 방향으로 판단하는가

이 파일은 이 저장소 구조에서 독특한 위치를 차지합니다. "제품 감각(product sense)"은 수치화하기 어렵고, 구체적인 기능 명세도 아니며, 기술적 규범도 아닙니다. 그러나 에이전트가 명확한 지시 없이 모호한 상황에 직면했을 때 **어떤 방향으로 판단을 내려야 하는가**를 알려주는 문서입니다.

에이전트는 종종 명세되지 않은 상황에 직면합니다. 두 가지 구현 방식이 기술적으로 동등할 때, 사용자 경험과 개발 편의가 충돌할 때, 빠른 해결책과 올바른 해결책 사이에서 선택해야 할 때. 이런 상황에서 에이전트에게 필요한 것은 규칙이 아니라 **판단 기준**입니다.

**다루는 내용:**

- 이 제품의 핵심 사용자가 누구이고 그들이 가장 중요하게 여기는 것이 무엇인지
- 이 제품이 경쟁 또는 대안과 구분되는 핵심 가치가 무엇인지
- 기능 충돌이 생겼을 때 무엇을 우선하는지의 원칙
- 이 제품에서 "좋은 경험"이란 어떤 것인지에 대한 기술

---

## 포지션

**"Design Intelligence, not Compliance Engine"**
**"우리는 설계자가 원하게 만드는 컨설턴트다"**

CAI는 법규 검토·인허가·BIM 자동화·구조 안전·시공비 산출 영역에 진입하지 않습니다.
독점 영역: 설계 언어·미학적 판단 / 브랜드→공간 번역 / 설계자 창의성 증폭.

## 의사결정 기준

### 기능 추가 여부

```
추가해도 되는가?
- 설계자의 창의성을 증폭시키는가? → Yes
- 법적 효력이나 기술적 보장을 암시하는가? → No
- 기존 노드의 핵심 기능을 희석시키는가? → No
- Phase-Lock 원칙을 지키는가? → Yes
```

### 범위 결정

노드 앱은 단일 목적입니다. 하나의 노드에 두 가지 핵심 기능을 넣지 않습니다.
기능이 추가되면 새 노드를 만들거나 기존 노드의 Knowledge Doc을 확장합니다.

## 타겟 세그먼트 우선순위

| 우선순위 | 세그먼트 | 핵심 Pain Point |
|----------|---------|-----------------|
| 1 (Primary) | 1인·중소 설계사 | 대형 사무소 대비 퀄리티 한계 |
| 2 (Secondary) | 브랜딩 에이전시 | 브랜드 감성 → 공간 번역 불가 |
| 3 (Entry) | 건축학생 | 실무 도구 접근성 부재 |
| 4 (Derived) | 건축주 | 설계 과정 블랙박스 |

## 법적 범위 원칙

- 법규·구조·시공 보장 문구 전면 배제
- 결과물에 법적 효력 없음 원칙 준수
- 이용약관에 서비스 범위 명시 필수
